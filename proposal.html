1 Project Overview

What we're building and why

Key Problem

  • Customer follow-ups managed via Excel & WhatsApp — no tracking
  • Sales data siloed across 10 counters using different billing systems
  • No mechanism to detect when a patient's medication cycle needs renewal
  • CSR team has no structured daily workload or ticketing
  • B2C online orders handled ad hoc with no status tracking

Proposed Solution

  • Aggregate daily sales & stock uploads from all counters
  • Auto-generate patient profiles and follow-up tickets from sales data
  • Structured CSR ticket board for daily follow-up operations
  • Online order creation and counter-based fulfillment flow
  • Admin visibility into CSR performance and sales patterns
This platform does not unify item master data across counters. It works as a coordination layer on top of existing systems.

3 Features Breakdown

6 modules · everything included

Module 01

Admin Dashboard

  • Master Item List upload (CSV/Excel)
  • Counter management (create/activate)
  • CSR user management & goal assignment
  • Platform-wide follow-up targets
  • Performance dashboard & SLA config
Module 02

Counter Upload Dashboard

  • Daily sales & stock CSV/Excel upload
  • Upload status calendar (Green/Yellow/Red)
  • Order notifications (Accept/Decline/Partial)
Module 03

Patient Profile System

  • Auto profile creation from sales uploads
  • Multi-counter profile support
  • Manual profile creation for online orders
  • Full purchase history tracking
  • Active / Inactive (60-day threshold)
Module 04

Follow-Up Ticket Engine

  • Tier 1 / Tier 2 ticket generation logic
  • Rule-based suppression to avoid overload
  • Auto-assign + admin override
  • Full lifecycle: Open → Resolved/Expired
  • SLA monitoring & escalation alerts
Module 05

CSR Ticket Board

  • Prioritized daily queue per agent
  • Filter, search, tag
  • Outcome logging (Interested/No Answer…)
  • Individual performance dashboard
Module 06

Online Order Flow

  • CSR-initiated order creation
  • Counter availability check
  • Accept / Decline per counter
  • Order status lifecycle
  • Follow-up scheduling for dispensed meds

8 Assumptions

Basis of this proposal

The following assumptions underpin scope and cost. Full Terms & Conditions will be reviewed and finalized at the next meeting prior to contract signing.

Delivery

  • Client provides counter data format samples within 3 days of kickoff
  • Client feedback within 2 business days per review cycle
  • Max 2 revision rounds per milestone
  • Scope changes via formal Change Request (CR)

Data

  • Sales and stock uploads treated independently — no cross-validation
  • Counter item codes and master codes remain non-unified
  • Admin-provided customer list is the initial source of truth
  • Upload file formats shared before upload module development begins

Operational

  • PharmaLife designates a primary point of contact
  • Counter users have basic internet access for file uploads
  • Physical dispatch and payment remain outside the platform
  • Automated notifications (WhatsApp/SMS) are out of scope for this phase

Infrastructure

  • Client funds staging and production environments from kickoff
  • Estimated server cost: NPR 15,000–30,000/month
  • Kush Tech provides deployment scripts; hosting managed by client
